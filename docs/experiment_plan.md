## 1. Базовые модели

### 1.1. SASRec-ID (классический SASRec с обучаемой таблицей эмбеддингов)

Классический SASRec. Модель обучается с функцией потерь sampled softmax и log-Q коррекцией.

Эта модель улавливает коллаборативные сигналы. Прямого доступа к аудио-содержимому треков у неё нет.

Роль в работе — бейзлайн и модель для основных экспериментов, где к ней добавляются семантические коды.

### 1.2. SASRec-Content (SASRec на готовых аудио-эмбеддингах из датасета)

Отличается от SASRec-ID только входом: вместо обучаемых эмбеддингов треков подаются фиксированные аудио-эмбеддинги из Yambda. Над аудио-эмбеддингами применяется обучаемый линейный слой, который приводит их к размерности, ожидаемой трансформером. Обучение происходит по той же схеме, что и для SASRec-ID.

Эта модель улавливает содержательные предпочтения пользователя — какие акустические свойства треков ему нравятся.

Роль в работе — источник эмбеддингов `z_u^{content}`, на котором затем обучается RQ-VAE (для получения семантических кодов).

### 1.3. Сжатие методом PCA

На матрице векторов `z_u^{content}` всех трейн пользователей обучается PCA. По обучающей матрице вычисляются среднее `μ` и матрица главных компонент `W`. Каждому пользователю соответствуют два представления:

- **Сжатый вектор** `z_{pca} = W_k^T · (z_u^{content} − μ)` (первые `k` главных компонент).
- **PCA-реконструкция** `z_u^{PCA-reconstruct} = W_k · z_{pca} + μ` размерности `d_model` — вектор в исходном пространстве, но с потерей информации за счёт отбрасывания компонент старше `k`.

Число сохраняемых компонент `k` — параметр, единого значения которого не фиксируется заранее. В разных экспериментах используются разные его значения:

- Для контроля «любое сжатие» как дополнительного признака (эксперимент 3.1) используется небольшое `k`, сохраняющее около половины исходной дисперсии.
- Для контроля равной ошибке восстановления (эксперимент 3.8) `k` подбирается так, чтобы средняя ошибка PCA-реконструкции `||z_u^{content} − z_u^{PCA-reconstruct}||²` на обучающих пользователях была максимально близка к средней ошибке восстановления RQ-VAE `||z_u^{content} − z_q||²`. Это позволяет честно сравнивать два способа сжатия при одинаковом объёме потерянной информации.

Роль в работе — контроль эффекта сжатия: позволяет отделить эффект семантической дискретизации от эффекта, который даёт просто уменьшение размерности эмбеддинга.

## 2. Получение семантических ID и их проверка

Здесь происходит построение самих семантических кодов и проверяется, что они несут содержательную информацию о пользователях. Без этих шагов последующие измерения влияния кодов на качество рекомендаций интерпретировать нельзя.

### 2.1. Обучение RQ-VAE для построения кодов

**Что делается.** Обучается стандартный RQ-VAE для получения семантических кодов. У RQ-VAE есть D уровней, на каждом уровне — своя таблица из K векторов.

Обучение проходит на векторах `z_u^{content}` обучающих пользователей. После обучения каждый пользователь получает код `(c_1, …, c_D)` длины D. Стартовая конфигурация — D = 4, K = 256.

**Зачем нужен.** Это сам способ получения семантических ID.

### 2.2. Подбор глубины кода D и размера кодовой книги K

**Что делается.** Абляции по D и K. 2.1 повторяется с разными значениями D при K фиксированном и с разными K при D фикс. Для каждой конфигурации измеряется ошибка восстановления и доля используемых кодов.

**Зачем нужен.** Выбрать одну рабочую конфигурацию (D, K), которая будет использоваться во всех основных экспериментах. Плюс нужно получить конфигурацию ёмкостью ~20 бит на пользователя для эксперимента 3.5.

### 2.3. Рекомендации по точному совпадению кодов (проверка содержательности)

**Что делается.** Никакого обучения, все вычисления — это поиск по готовым таблицам. Для каждого пользователя из теста берётся его код, вычисленный по его истории. Среди трейн пользователей находятся те, у кого код совпадает полностью, — это группа «соседей» данного пользователя. Рекомендация — K самых популярных треков внутри этой группы.

**Зачем нужен.** Проверить, что код действительно объединяет похожих пользователей. Если такая рекомендация заметно лучше случайной — коды несут содержательную информацию.

### 2.4. Рекомендации по перестановленным кодам (контроль содержательности)

**Что делается.** То же самое, что в 2.3, но коды настоящих пользователей случайно перемешиваются между собой: каждому пользователю приписывается настоящий код другого случайно выбранного обучающего пользователя. При этом маргинальное распределение кодов сохраняется — разрушается только связь «этот код — этот пользователь». Рекомендация строится по той же схеме, что в 2.3

**Зачем нужен.** Отделить эффект «разбиение пользователей на группы само по себе помогает» от эффекта «коды несут содержательную информацию именно об этом пользователе».

### 2.5. (ДОП) Рекомендации методом k ближайших соседей и по популярности

**Что делается.** Два дополнительных простых способа рекомендаций. Первый: для каждого пользователя из теста находятся k ближайших пользователей из трейна по косинусной близости в пространстве `z_u^{content}`, из их историй берутся популярные треки. Второй: просто K самых популярных треков в трейне — без персонализации.

**Зачем нужен.** Верхняя и нижняя точки отсчёта для эксперимента 2.3. Рекомендации по ближайшим соседям показывают, какого качества можно достичь через близость в непрерывном пространстве (верхняя граница). Рекомендации по популярности — нижняя граница.

## 3. Основные эксперименты: измерение влияния семантических ID на качество рекомендаций

Здесь проверяется, как именно коды влияют на качество рекомендаций. Все основные эксперименты построены на объединении кодов с рекомендательной моделью и на проверке, какой вклад даёт код при разных способах его использования.

### 3.1. (ОСН) Объединение представлений: коды как обучаемые признаки

**Что делается.** Берётся уже обученный SASRec-ID (пункт 1.1). На этапе применения модели её итоговый вектор пользователя `z_u^{ID}` (коллаборативный сигнал) объединяется с семантическим id пользователя. Для этого используется таблица эмбеддингов: для каждого из D уровней своя таблица размера K. Итоговое кодовое представление для семантического id это сумма векторов выбранных из этих таблиц по кодам пользователя:  `код_emb = E_1[c_1] + E_2[c_2] + … + E_D[c_D]`. Затем `z_u^{ID}` и `код_emb` объединяются через линейный слой, результат используется для предсказания следующего трека.

Обучается только новый линейный слой объединения и таблицы `E_d`. Сам SASRec-ID и кодовые книги RQ-VAE остаются без изменений.

**Контроли к этому эксперименту:**
- Такая же схема с перестановленными кодами: коды настоящих пользователей случайно перемешиваются между собой. Это отделяет эффект «коды содержательны именно для этого пользователя» от эффекта «дополнительная обучаемая ёмкость помогает сама по себе» и от эффекта «неравномерное распределение индексов в таблице эмбеддингов даёт преимущество». Равномерно случайные коды не подходят как контроль, потому что у них распределение индексов равномерное, а у настоящих кодов RQ-VAE — скорее всего неравномерное.
- Такая же схема, где вместо `код_emb` используется сжатый вектор `z_{pca}` из пункта 1.3. Контроль отделяет «дискретная структура полезна» от «любое содержательное дополнительное представление полезно».

**Зачем нужен.** Ответ на вопрос: улучшают ли коды качество рекомендаций в сравнении с моделью без них. Если объединение значимо лучше обычной SASRec-ID и значимо лучше варианта с перестановленными кодами — коды несут полезный сигнал, дополняющий коллаборативную модель.

### 3.2. (ОСН) Объединение представлений: коды как готовые семантические векторы из RQ-VAE (пара к 3.1)

**Что делается.** Повторяется схема эксперимента 3.1, но вместо обучаемых таблиц эмбеддингов `E_d` кодовое представление пользователя берётся напрямую из кодовых книг RQ-VAE, обученных в разделе 2.1: `z_q = codebook_1[c_1] + codebook_2[c_2] + … + codebook_D[c_D]`

Эти векторы не обучаются — они фиксируются как есть после 2.1. Затем `z_u^{ID}` и `z_q` объединяются через обучаемый линейный слой, результат используется для предсказания следующего трека. Обучается только линейный слой объединения; SASRec-ID, RQ-VAE и кодовые книги остаются без изменений.

**Зачем нужен.** В эксперименте 3.1 таблицы `E_d` инициализируются случайно и обучаются под задачу рекомендаций. Они не обязаны сохранять семантическую структуру, заложенную в кодах. Например, два пользователя с близкими кодами `(3, 17, 1, 3)` и `(3, 17, 1, 4)` могут получить совершенно разные векторы, потому что строки `E_4[3]` и `E_4[4]` обучаются независимо и между ними нет никакой связи — это просто два разных категориальных индекса.

Сравнение 3.1 и 3.2 изолирует вопрос: какой вклад в качество рекомендаций даёт именно семантика кодов, зафиксированная при обучении RQ-VAE, а какой — обучаемая под задачу интерпретация кодов как категориальных индексов.

### 3.3. (ДОП) Варьирование ёмкости кодов в экспериментах 3.1 и 3.2

**Что делается.** Абляции по D и K. Эксперименты 3.1 и 3.2 повторяются с разными значениями глубины кода D при фиксированном K, а затем с разными размерами кодовой книги K при фиксированном D.

**Зачем нужен.** Даёт две параллельные кривые «сколько информации в коде — какой эффект в рекомендациях»: одну для обучаемой интерпретации кодов (3.1), вторую для фиксированной семантической (3.2). Это превращает ответ из бинарного («влияют ли») в количественный («как именно влияют»).

### 3.4. (ДОП) Вклад каждого уровня иерархии по отдельности

**Что делается.** Обучается одна модель по схеме эксперимента 3.1 с максимальной глубиной кода (например, D = 8). При обучении применяется случайное маскирование уровней: на каждом шаге случайное подмножество уровней зануляется — это не даёт модели сколлапсировать всю информацию в несколько первых уровней и заставляет её использовать все уровни иерархии.

После обучения модель применяется в нескольких режимах: сначала используется только верхний уровень кода (`код_emb = E_1[c_1]`), затем два верхних (`E_1[c_1] + E_2[c_2]`), и так далее до полного использования всех уровней.

**Зачем нужен.** Проверить заявленную иерархическую природу кода через влияние на рекомендации: верхние уровни должны давать основной вклад в качество (они кодируют широкие предпочтения), каждый следующий — уточнение.

### 3.5. (ОСН) Контроль с кластеризацией методом k-means

**Что делается.** Сравниваются два способа построения замороженной таблицы векторов пользователя при одинаковом битовом бюджете: через RQ-VAE (по схеме эксперимента 3.2) и через кластеризацию k-means. В обеих версиях линейный слой объединения обучаемый, `z_u^{ID}` берётся из SASRec-ID, таблица векторов замороженная. Число обучаемых параметров в обеих версиях совпадает.

Конкретная конфигурация (D, K) и соответствующий ей битовый бюджет фиксируются после завершения разделов 2.2 и 3.3. Критерий выбора — наибольший битовый бюджет, при котором k-means с `2^бит` кластеров остаётся разумным и технически реализуемым.

**Процедура:**

1. На векторах `z_u^{content}` трейн пользователей обучается k-means с числом кластеров, равным выбранному `2^бит`. Каждому пользователю приписывается номер его ближайшего кластера в качестве альтернативного кода.
2. Строятся две версии замороженной таблицы векторов пользователя:
   - **RQ-VAE-версия.** Вектор пользователя — сумма из кодовых книг: `z_q = codebook_1[c_1] + … + codebook_D[c_D]`, где кодовые книги взяты из уже обученного в 2.2 RQ-VAE выбранной конфигурации.
   - **k-means-версия.** Вектор пользователя — центроид его кластера.
3. В обеих версиях таблица векторов замораживается. Обучается только линейный слой объединения `z_u^{ID}` и вектора пользователя. В остальном — как в эксперименте 3.2.
4. Измеряются метрики на тесте и сравниваются между двумя версиями.

**Зачем нужен.** Отделить вклад именно иерархической квантизации RQ-VAE от вклада «любого содержательного разбиения пользователей на группы по их акустическим предпочтениям». Случайные коды не сохраняют близость пользователей, а k-means сохраняет — поэтому k-means является честным конкурентом. Число обучаемых параметров в обеих версиях совпадает. Если RQ-VAE-версия значимо превосходит k-means — значит, именно иерархическая residual-квантизация даёт семантически лучшие векторы, чем плоская кластеризация. Если они неразличимы — наблюдаемый эффект объясняется просто разбиением пользователей на содержательные группы, а сама процедура обучения RQ-VAE дополнительного вклада не даёт.

### 3.6. (ОСН) Подмена кода пользователя

**Что делается.** Используется уже обученная модель из эксперимента 3.1. Никакого дополнительного обучения нет. Для каждого пользователя из теста рекомендации генерируются три раза с разными вариантами кода:
- (а) настоящий код пользователя;
- (б) код ближайшего соседа пользователя в пространстве `z_u^{content}` (похожий вкус);
- (в) случайный код другого обучающего пользователя (случайный чужой вкус).

Измеряются метрики в каждом варианте.

**Зачем нужен.** Количественно оценить, насколько рекомендации чувствительны к конкретному коду. Это прямой операциональный смысл «влияния»: если бы коды ни на что не влияли, подмена не должна была бы менять метрики. Ожидаемый порядок: (а) > (б) > (в). Если (а) ≈ (в) — модель игнорирует код, а наблюдаемое улучшение в эксперименте 3.1 объясняется дополнительной обучаемой ёмкостью, а не содержанием кода.

### 3.7. (ДОП) Рекомендации только по семантическому коду пользователя

**Что делается.** Обучается модель матричной факторизации, в которой представление пользователя формируется исключительно из его семантического кода, без использования истории прослушиваний. Никакой другой модели здесь нет.

Представление пользователя — сумма векторов из D обучаемых таблиц эмбеддингов, выбранных по коду пользователя: `z_u = E_1[c_1] + E_2[c_2] + E_3[c_3] + E_4[c_4]`

Каждая таблица `E_d` имеет размер K × h (то есть по одному вектору размерности h для каждого возможного значения кода на d-ом уровне) и инициализируется случайно. Представление трека — обучаемый эмбеддинг `v_i` размерности h, как в SASRec-ID. Оценка релевантности трека `i` пользователю `u` — скалярное произведение `score(u, i) = z_u · v_i`.

Обучение проводится на тех же парах «пользователь — следующий прослушанный трек», что и у SASRec, с той же функцией потерь (sampled softmax с поправкой log-Q). Обучаются таблицы `E_1, …, E_D` и эмбеддинги треков. Разница с SASRec-ID в том, что здесь модель не видит историю пользователя как последовательность — единственным источником информации о пользователе является его статичный код.

**Зачем нужен.** Проверить, насколько информативен код **сам по себе**, без опоры на историю.

### 3.8. (ОСН) Замена непрерывного вектора на восстановление из кодов

**Что делается.** Вместо вектора из SASRec-Content (пункт 1.2) берётся вектор, восстановленный из семантических кодов: `z_q = codebook_1[c_1] + … + codebook_D[c_D]` (суммируются векторы, выбранные из кодовых книг RQ-VAE по кодам пользователя).

**Контроли к этому эксперименту:**
- Замена на вектор, восстановленный из перестановленных кодов: коды пользователей случайно перемешиваются между собой, и для каждого пользователя вектор восстанавливается по коду другого, случайно выбранного пользователя. Это сохраняет распределение восстановленных векторов и разрушает только соответствие «пользователь — его вектор».
- Замена на PCA-реконструкцию `z_u^{PCA-reconstruct}` из пункта 1.3: исходный вектор `z_u^{content}` проецируется на первые `k` главных компонент и обратно восстанавливается в пространство размерности `d_model`. Число компонент `k` подбирается так, чтобы средняя ошибка PCA-реконструкции была сопоставима со средней ошибкой восстановления RQ-VAE. При таком подборе оба контроля теряют примерно одинаковое количество информации, и сравнение изолирует именно разницу между способами представления: иерархические дискретные коды против непрерывной линейной проекции.

**Зачем нужен.** Проверить, насколько дискретизация сохраняет полезную информацию: «на сколько коды ухудшают качество при полной замене непрерывного представления». Если качество падает незначительно — код является практически равноценной заменой непрерывного вектора. Если падает заметно — коды являются только дополнением, но не заменой.
