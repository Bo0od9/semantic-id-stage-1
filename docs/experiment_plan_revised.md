# План экспериментов: влияние семантических ID пользователя на качество рекомендаций

## Главный исследовательский вопрос

**Как семантические ID пользователя влияют на качество рекомендательной системы?**

Под семантическим ID пользователя понимается короткая последовательность дискретных кодов `(c_1, c_2, c_3, c_4)`, каждый из которых — целое число от 0 до K−1 (при K = 256 — один байт). Такой ID должен заменить или дополнить непрерывный векторный портрет пользователя и быть пригодным для использования в рекомендательной системе.

Работа строится на датасете Yambda (история прослушиваний треков пользователями Яндекс Музыки и предобученные аудио-эмбеддинги треков). Задача — предсказание следующего трека по истории прослушиваний. Основные метрики — Recall@K, NDCG@K и HitRate@K при K ∈ {10, 100}.

Все эксперименты запускаются на трёх случайных начальных значениях генератора (42, 43, 44), результаты приводятся как среднее ± стандартное отклонение. Значимость различий оценивается по парному критерию Уилкоксона с поправкой Бонферрони.

## 1. Базовые модели

Две версии модели SASRec и одна линейная модель сжатия.

### 1.1. SASRec-ID (классический SASRec с обучаемой таблицей эмбеддингов)

Классический SASRec: каждому треку каталога соответствует обучаемый векторный эмбеддинг, который подстраивается в процессе обучения. На вход трансформера подаётся последовательность таких эмбеддингов, соответствующая истории пользователя. Модель обучается на обучающей части данных с функцией потерь sampled softmax и поправкой log-Q.

Эта модель улавливает коллаборативные закономерности. Прямого доступа к аудио-содержимому треков у неё нет.

Роль в работе — коллаборативная точка отсчёта и среда для основных экспериментов, где к ней добавляются семантические коды.

### 1.2. SASRec-Content (SASRec на готовых аудио-эмбеддингах из датасета)

Отличается от SASRec-ID только входом: вместо обучаемых эмбеддингов треков подаются фиксированные аудио-эмбеддинги из Yambda. Над аудио-эмбеддингами применяется небольшой обучаемый линейный слой, который приводит их к размерности, ожидаемой трансформером. Обучение происходит по той же схеме, что и для SASRec-ID.

Эта модель улавливает содержательные предпочтения пользователя — какие акустические свойства треков ему нравятся.

Роль в работе — содержательная точка отсчёта и источник непрерывного вектора `z_u^{содерж.}`, на котором затем обучается RQ-VAE (для получения семантических кодов).

### 1.3. Сжатие методом PCA

На матрице векторов `z_u^{содерж.}` всех обучающих пользователей обучается PCA. По обучающей матрице вычисляются среднее `μ` и матрица главных компонент `W` (столбцы — собственные векторы ковариационной матрицы, отсортированные по убыванию собственных значений). Каждому пользователю соответствуют два представления:

- **Сжатый вектор** `z_{pca} = W_k^T · (z_u^{содерж.} − μ)` размерности `k` (первые `k` главных компонент).
- **PCA-реконструкция** `z_u^{PCA-восст.} = W_k · z_{pca} + μ` размерности `d_model` — вектор в исходном пространстве, но с потерей информации за счёт отбрасывания компонент старше `k`.

Число сохраняемых компонент `k` — параметр, единого значения которого не фиксируется заранее. В разных экспериментах используются разные его значения:

- Для контроля «любое сжатие» как дополнительного признака (эксперимент 3.1) используется небольшое `k`, сохраняющее около половины исходной дисперсии (практически 10–15 компонент).
- Для контроля равной ошибке восстановления (эксперимент 3.8) `k` подбирается так, чтобы средняя ошибка PCA-реконструкции `‖z_u^{содерж.} − z_u^{PCA-восст.}‖²` на обучающих пользователях была максимально близка к средней ошибке восстановления RQ-VAE `‖z_u^{содерж.} − z_q‖²`. Это позволяет честно сравнивать два способа сжатия при одинаковом объёме потерянной информации.

Роль в работе — контроль «любое сжатие»: позволяет отделить эффект от дискретизации как таковой от эффекта, который даёт просто уменьшение размерности непрерывного представления.

### 1.4. Варианты извлечения непрерывного вектора пользователя из SASRec-Content

**Что делается.** У SASRec-Content скрытое состояние трансформера определено для каждой позиции истории пользователя. Непрерывный вектор пользователя `z_u^{содерж.}` может быть получен из этих состояний двумя способами:

- `z_u^{содерж., last}` — скрытое состояние в позиции последнего события в истории. Это стандартный способ извлечения в SASRec: именно этот вектор используется во время обучения для предсказания следующего трека и в стандартном протоколе оценивания SASRec.
- `z_u^{содерж., mean}` — среднее скрытых состояний по всем валидным позициям истории. Последние события не имеют перевеса над более ранними; вектор отражает весь профиль прослушиваний как единое целое.

Обе версии получаются одним проходом уже обученной SASRec-Content и не требуют дополнительного обучения. Они сравниваются по двум критериям:

- Стандартные рекомендательные метрики Recall@K, NDCG@K, HitRate@K на тестовой части — через скалярное произведение `z_u^{содерж.} · e_i` и ранжирование по каталогу.
- Содержательная однородность групп пользователей: пользователи кластеризуются в пространстве `z_u^{содерж.}` (например, через k-means с небольшим числом кластеров), внутри каждой группы измеряется однородность акустических предпочтений через агрегированные характеристики прослушанных треков или, при наличии, через жанровую разметку.

На основании этих измерений одна из двух версий фиксируется как основной `z_u^{содерж.}` и используется во всех последующих разделах (обучение RQ-VAE в 2.1, основные эксперименты 3.1, 3.2, 3.8). После выбора вариант фиксируется до перехода к разделу 2 и далее не меняется.

Возможные исходы:

- **mean заметно лучше last по обоим критериям.** Основной способ — mean. Сильный аргумент в пользу того, что для семантического представления пользователя полезнее аггрегация всей истории, чем prediction-query последнего токена.
- **last лучше по рекомендательным метрикам, mean — по однородности групп.** Возникает компромисс между качеством «в ближайшей точке» и качеством семантической интерпретации. Решение принимается в пользу mean, поскольку главный исследовательский вопрос — про описание предпочтений, и явно отражается в тексте диплома как осознанный выбор в пользу интерпретируемости.
- **Обе версии сопоставимы по обоим критериям.** Используется last как более стандартный вариант, совместимый с классическим обучением SASRec.

**Зачем нужен.** Главный исследовательский вопрос формулируется про **описание музыкальных предпочтений** пользователя через коды, а не про прогноз ближайшего трека. Стандартное извлечение last-token оптимизировано под next-item prediction и может непропорционально зависеть от последних событий истории. Среднее по истории даёт более устойчивое агрегированное представление, но может хуже работать в стандартных рекомендательных метриках, потому что оно не натренировано под next-item. Явный выбор между ними до перехода к RQ-VAE гарантирует, что непрерывный вектор, подаваемый в последующие шаги, согласован с целью работы, а не унаследован автоматически из next-item prediction.

## 2. Предварительные шаги и проверочные эксперименты

Здесь происходит построение самих семантических кодов и проверяется, что они несут содержательную информацию о пользователях. Без этих шагов последующие измерения влияния кодов на качество рекомендаций интерпретировать нельзя.

### 2.1. Обучение RQ-VAE для построения кодов

**Что делается.** RQ-VAE — это метод, превращающий непрерывный вектор в последовательность из D целых чисел через жадную иерархическую квантизацию. У RQ-VAE есть D уровней, на каждом уровне — своя таблица из K векторов (кодовая книга). Для данного вектора `z_u^{содерж.}` на первом уровне выбирается ближайший вектор из первой кодовой книги, его номер записывается как `c_1`. Разница между исходным вектором и выбранным (остаток) квантуется на втором уровне из второй кодовой книги, номер выбранного вектора — `c_2`. Процесс повторяется до D-го уровня.

Обучаются сами кодовые книги — так, чтобы сумма выбранных векторов по всем уровням как можно точнее восстанавливала исходный `z_u^{содерж.}`. Функция потерь состоит из двух частей: ошибка восстановления и штраф за удаление кодовых векторов от исходного вектора. Для борьбы с вырождением (когда часть кодов не используется) применяется скользящее среднее при обновлении кодовых книг и принудительный сброс неиспользуемых кодов.

Обучение проходит на векторах `z_u^{содерж.}` обучающих пользователей. После обучения каждый пользователь получает код `(c_1, …, c_D)` длины D. Стартовая конфигурация — D = 4, K = 256 (32 бита на пользователя).

**Зачем нужен.** Это сам способ получения семантических ID, без него проводить эксперименты по влиянию кодов невозможно. Дополнительно на этом шаге отслеживается доля используемых кодов на каждом уровне — она должна превышать 50 %, иначе результаты последующих экспериментов будут искажены вырожденной кодовой книгой.

### 2.2. Подбор глубины кода D и размера кодовой книги K

**Что делается.** Эксперимент 2.1 повторяется с разными значениями D ∈ {2, 4, 6, 8} при K = 256 и с разными K ∈ {32, 64, 256, 1024} при D = 4. Для каждой конфигурации измеряется ошибка восстановления и доля используемых кодов. Конфигурация D = 4, K = 32 (20 бит на пользователя) включена специально как возможный кандидат для использования в эксперименте 3.5 (контроль с k-means) — она расширяет нижнюю границу бюджета на один шаг.

**Зачем нужен.** Выбрать одну рабочую конфигурацию (D, K), которая будет использоваться во всех основных экспериментах. Без предварительного подбора есть риск получить отрицательный результат просто из-за неудачной глубины или размера кодовой книги. Выбранная конфигурация фиксируется до перехода к разделу 3 и далее не меняется.

### 2.3. Рекомендации по точному совпадению кодов (проверка содержательности)

**Что делается.** Никакого обучения; все вычисления — это поиск по готовым таблицам. Для каждого пользователя из проверочного набора берётся его код, вычисленный по истории до точки отсечения. Среди обучающих пользователей находятся те, у кого код совпадает полностью, — это группа «соседей» данного пользователя. Рекомендация — K самых популярных треков внутри этой группы.

**Зачем нужен.** Проверить, что код действительно объединяет похожих пользователей. Если такая рекомендация заметно лучше случайной — коды несут содержательную информацию и дальнейшие эксперименты имеют смысл. Если нет — обучение RQ-VAE нужно пересмотреть до перехода к разделу 3.

### 2.4. Рекомендации по перестановленным кодам (контроль содержательности)

**Что делается.** То же самое, что в 2.3, но коды настоящих пользователей случайно перемешиваются между собой: каждому пользователю приписывается настоящий код другого случайно выбранного обучающего пользователя. При этом маргинальное распределение кодов (какие значения встречаются чаще, какие реже) сохраняется — разрушается только связь «этот код — этот пользователь». Рекомендация строится по той же схеме, что в 2.3, — K самых популярных треков среди пользователей с совпавшим кодом.

**Зачем нужен.** Отделить эффект «разбиение пользователей на группы само по себе помогает» от эффекта «коды несут содержательную информацию **именно об этом** пользователе». Перестановка — более строгий контроль, чем равномерно случайные коды: она сохраняет все статистические свойства кодового пространства (размер групп, их перекрытие) и разрушает только соответствие «пользователь — его настоящий код». Если результат 2.3 заметно выше результата 2.4 — коды действительно содержательны в смысле персонализации, а не просто удачно разбивают пользователей на группы.

### 2.5. Рекомендации методом k ближайших соседей и по популярности

**Что делается.** Два дополнительных простых способа рекомендаций. Первый: для каждого проверочного пользователя находятся k ближайших обучающих пользователей по косинусной близости в пространстве `z_u^{содерж.}`, из их историй берутся популярные треки. Второй: просто K самых популярных треков в обучающей части — без персонализации.

**Зачем нужен.** Верхняя и нижняя точки отсчёта для эксперимента 2.3. Рекомендации по ближайшим соседям показывают, какого качества можно достичь через близость в непрерывном пространстве (верхняя граница). Рекомендации по популярности — нижняя граница: любая персонализированная модель должна её превосходить. Ожидаемый порядок качества: соседи по непрерывному вектору ≥ рекомендации по точному совпадению кода > рекомендации по перестановленному коду ≥ рекомендации по популярности.

## 3. Основные эксперименты: измерение влияния семантических ID на качество рекомендаций

Ядро работы. Здесь проверяется, как именно коды влияют на качество рекомендаций. Все основные эксперименты построены на объединении кодов с рекомендательной моделью и на проверке, какой вклад даёт код при разных способах его использования.

### 3.1. (ОСН) Объединение представлений: коды как обучаемые признаки (главный эксперимент)

**Что делается.** Берётся уже обученная SASRec-ID (пункт 1.1). На этапе применения модели её итоговый вектор пользователя `z_u^{ID}` (коллаборативный портрет) объединяется с кодовым представлением пользователя. Кодовое представление устроено так: для каждого из D уровней заводится своя обучаемая таблица эмбеддингов размера K (то есть D новых таблиц), инициализируемая случайно. Кодовое представление — это сумма векторов, выбранных из этих таблиц по кодам пользователя: `код_emb = E_1[c_1] + E_2[c_2] + … + E_D[c_D]`. Затем `z_u^{ID}` и `код_emb` объединяются через линейный слой, результат используется для предсказания следующего трека.

Обучается только новый линейный слой объединения и таблицы `E_d`. Сама SASRec-ID и кодовые книги RQ-VAE остаются без изменений. Обучение идёт на обучающем наборе, метрики измеряются на тестовом.

**Контроли к этому эксперименту:**
- Такая же схема с **перестановленными кодами**: коды настоящих пользователей случайно перемешиваются между собой (каждому пользователю приписывается настоящий код другого случайно выбранного пользователя). Маргинальное распределение кодов и размер соответствующих групп сохраняются — разрушается только связь «этот код — этот пользователь». Это отделяет эффект «коды содержательны именно для этого пользователя» от эффекта «дополнительная обучаемая ёмкость помогает сама по себе» и от эффекта «неравномерное распределение индексов в таблице эмбеддингов даёт преимущество». Равномерно случайные коды не подходят как контроль, потому что у них распределение индексов равномерное, а у настоящих кодов RQ-VAE — заметно неравномерное, и это различие само по себе могло бы объяснить разницу в качестве.
- Такая же схема, где вместо `код_emb` используется сжатый вектор `z_{pca}` из пункта 1.3 (в этом эксперименте — с `k` порядка 10–15 компонент, соответствующим половине сохранённой дисперсии). Здесь размерность `z_{pca}` не обязана совпадать с `d_model`, потому что вектор подаётся не напрямую в SASRec-Content, а в обучаемый линейный слой объединения, который принимает вход произвольной размерности. Контроль отделяет «дискретная структура полезна» от «любое содержательное дополнительное представление полезно».

**Зачем нужен.** Прямой ответ на главный вопрос в самой простой форме: улучшают ли коды качество рекомендаций в сравнении с моделью без них. Если объединение значимо лучше обычной SASRec-ID и значимо лучше варианта с перестановленными кодами — коды несут полезный сигнал, дополняющий коллаборативную модель.

### 3.2. (ОСН) Объединение представлений: коды как готовые семантические векторы из RQ-VAE (пара к 3.1)

**Что делается.** Повторяется схема эксперимента 3.1, но вместо обучаемых таблиц эмбеддингов `E_d` кодовое представление пользователя берётся напрямую из кодовых книг RQ-VAE, обученных в разделе 2.1:

```
z_q = codebook_1[c_1] + codebook_2[c_2] + … + codebook_D[c_D]
```

Эти векторы **не обучаются** — они фиксируются как есть после раздела 2.1. Затем `z_u^{ID}` и `z_q` объединяются через обучаемый линейный слой, результат используется для предсказания следующего трека. Обучается только линейный слой объединения; SASRec-ID, RQ-VAE и кодовые книги остаются без изменений. Эксперимент существенно дешевле 3.1 и по числу обучаемых параметров, и по времени обучения.

**Зачем нужен и почему именно пара к 3.1.** В эксперименте 3.1 таблицы `E_d` инициализируются случайно и обучаются под задачу рекомендаций. Они не обязаны сохранять семантическую структуру, заложенную в кодах. Например, два пользователя с близкими кодами `(3, 17, 1, 3)` и `(3, 17, 1, 4)` могут получить совершенно разные векторы, потому что строки `E_4[3]` и `E_4[4]` обучаются независимо и между ними нет никакой связи — это просто два разных категориальных индекса.

В эксперименте 3.2 используются векторы RQ-VAE, устроенные принципиально иначе: близкие по смыслу пользователи имеют близкие реконструкции **по построению**. Верхние уровни `codebook_1[3] + codebook_2[17] + codebook_3[1]` у обоих пользователей совпадают, а отличающиеся на последнем уровне `codebook_4[3]` и `codebook_4[4]` — это маленькие остатки, которые по определению residual quantization имеют небольшую норму. Семантическая близость кодов напрямую переводится в геометрическую близость их векторного представления.

Сравнение 3.1 и 3.2 изолирует принципиальный вопрос: какой вклад в качество рекомендаций даёт **именно семантика кодов**, зафиксированная при обучении RQ-VAE, а какой — **обучаемая под задачу интерпретация** кодов как категориальных индексов. Возможные исходы:

- **3.1 заметно лучше 3.2.** Семантической структуры RQ-VAE недостаточно; модели нужна возможность переинтерпретировать коды под задачу. Полезность RQ-VAE — в том, что он даёт осмысленное разбиение пользователей на группы, а не в геометрии самих кодовых векторов.
- **3.2 не хуже 3.1.** Семантика RQ-VAE сама по себе достаточна; обучаемая интерпретация не добавляет существенного. Это сильный результат: дополнительные обучаемые параметры не нужны, коды переносимы и работают как готовое плотное семантическое представление пользователя.
- **3.1 ≈ 3.2.** Оба подхода работают одинаково хорошо. Практический вывод: предпочтительнее более лёгкий вариант 3.2 (меньше обучаемых параметров), исследовательски оба варианта равноправны.

### 3.3. (ОСН) Варьирование ёмкости кодов в экспериментах 3.1 и 3.2

**Что делается.** Эксперименты 3.1 и 3.2 повторяются с разными значениями глубины кода D ∈ {2, 4, 6, 8} при фиксированной K, а затем с разными размерами кодовой книги K ∈ {32, 64, 256, 1024} при фиксированной D. Для каждой конфигурации и каждой из двух схем (обучаемые таблицы и фиксированные векторы RQ-VAE) фиксируются метрики рекомендаций.

**Зачем нужен.** Даёт две параллельные кривые «сколько информации в коде — какой эффект в рекомендациях»: одну для обучаемой интерпретации кодов (3.1), вторую для фиксированной семантической (3.2). Это превращает ответ из бинарного («влияют ли») в количественный («как именно влияют»), что и требуется главным вопросом. Параллельное наличие двух кривых позволяет увидеть, одинаково ли обе схемы используют дополнительную ёмкость: возможно, у 3.2 есть плато при некоторой D (больше семантики из RQ-VAE извлечь не удаётся), а у 3.1 рост продолжается (обучаемые таблицы могут использовать ёмкость как дополнительные категориальные признаки). Форма кривых сама по себе — содержательный результат.

### 3.4. (ДОП) Вклад каждого уровня иерархии по отдельности

**Что делается.** Обучается одна модель по схеме эксперимента 3.1 с максимальной глубиной кода (например, D = 8). При обучении применяется случайное маскирование уровней: на каждом шаге случайное подмножество уровней зануляется — это не даёт модели сколлапсировать всю информацию в несколько первых уровней и заставляет её использовать все уровни иерархии.

После обучения модель применяется в нескольких режимах: сначала используется только верхний уровень кода (`код_emb = E_1[c_1]`), затем два верхних (`E_1[c_1] + E_2[c_2]`), и так далее до полного использования всех уровней. Для каждого режима измеряются метрики рекомендаций.

**Зачем нужен.** Проверить заявленную иерархическую природу кода через влияние на рекомендации: верхние уровни должны давать основной вклад в качество (они кодируют широкие предпочтения), каждый следующий — уточнение. Это самый прямой эксперимент для проверки того, что иерархия кода не просто формально присутствует, а реально отражается на качестве рекомендаций.

### 3.5. (ОСН) Контроль с кластеризацией методом k-means

**Что делается.** Сравниваются два способа построения замороженной таблицы векторов пользователя при одинаковом битовом бюджете: через RQ-VAE (по схеме эксперимента 3.2) и через плоскую кластеризацию k-means. В обеих версиях линейный слой объединения обучаемый, `z_u^{ID}` берётся из SASRec-ID, таблица векторов замороженная. Число обучаемых параметров в обеих версиях совпадает точно (только слой объединения) — это принципиально для корректного сравнения.

Конкретная конфигурация (D, K) и соответствующий ей битовый бюджет фиксируются после завершения разделов 2.2 и 3.3. Критерий выбора — наибольший битовый бюджет, при котором плоский k-means с `2^бит` кластеров остаётся технически реализуемым за разумное время (менее суток на рабочей машине через MiniBatchKMeans), и при котором соответствующая конфигурация RQ-VAE показала в 3.3 разумное качество по сравнению с рабочей 32-битной конфигурацией. Ожидаемый диапазон бюджета — от 16 до примерно 22 бит; возможные конфигурации RQ-VAE — D = 2, K = 256 (16 бит) или D = 4, K = 32 (20 бит).

**Процедура:**

1. На векторах `z_u^{содерж.}` обучающих пользователей обучается k-means с числом кластеров, равным выбранному `2^бит`, через MiniBatchKMeans. Каждому пользователю приписывается номер его ближайшего кластера в качестве альтернативного кода.
2. Строятся две версии замороженной таблицы векторов пользователя:
   - **RQ-VAE-версия.** Вектор пользователя — сумма из кодовых книг: `z_q = codebook_1[c_1] + … + codebook_D[c_D]`, где кодовые книги взяты из уже обученного в 2.2 RQ-VAE выбранной конфигурации.
   - **k-means-версия.** Вектор пользователя — центроид его кластера.
3. В обеих версиях таблица векторов замораживается. Обучается только линейный слой объединения `z_u^{ID}` и вектора пользователя. В остальном — как в эксперименте 3.2.
4. Измеряются метрики на тестовой части и сравниваются между двумя версиями.

**Зачем нужен.** Отделить вклад именно иерархической квантизации RQ-VAE от вклада «любого содержательного разбиения пользователей на группы по их акустическим предпочтениям». Случайные коды не сохраняют близость пользователей, а k-means сохраняет — поэтому k-means является честным конкурентом. Число обучаемых параметров в обеих версиях совпадает точно, поэтому разница в качестве (если она есть) относится исключительно к способу построения таблицы векторов. Если RQ-VAE-версия значимо превосходит k-means-версию — именно иерархическая residual-квантизация даёт семантически лучшие векторы, чем центроиды плоской кластеризации. Если они неразличимы — наблюдаемый эффект объясняется просто разбиением пользователей на содержательные группы, а сама процедура обучения RQ-VAE дополнительного вклада не даёт.

**Ограничения:**

- Сравнение проводится не на целевых 32 битах, а на меньшем битовом бюджете — это технический предел плоского k-means. Если на этом бюджете разницы между RQ-VAE и k-means нет, это слабый, но разумный аргумент в пользу того, что её не будет и на 32 битах; если разница есть — она, скорее всего, сохраняется и при увеличении бюджета. Строгим доказательством для 32 бит это не является, и это ограничение явно отражается в тексте диплома.
- Эксперимент намеренно не сравнивается со схемой 3.1 (обучаемые таблицы), поскольку у k-means таблица замороженная. Для честного равенства числа обучаемых параметров контроль привязан к схеме 3.2 (обе версии используют замороженные таблицы).

### 3.6. (ОСН) Подмена кода пользователя

**Что делается.** Используется уже обученная модель из эксперимента 3.1. Никакого дополнительного обучения нет. Для каждого пользователя проверочного набора рекомендации генерируются три раза с разными вариантами кода:
- (а) настоящий код пользователя;
- (б) код ближайшего соседа пользователя в пространстве `z_u^{содерж.}` (похожий вкус);
- (в) случайный код другого обучающего пользователя (случайный чужой вкус).

Измеряются метрики качества в каждом варианте.

**Зачем нужен.** Количественно оценить, насколько рекомендации чувствительны к конкретному коду. Это прямой операциональный смысл «влияния»: если бы коды ни на что не влияли, подмена не должна была бы менять метрики. Ожидаемый порядок: (а) > (б) > (в). Если (а) ≈ (в) — модель игнорирует код, а наблюдаемое улучшение в эксперименте 3.1 объясняется дополнительной обучаемой ёмкостью, а не содержанием кода.

### 3.7. (ДОП) Рекомендации только по семантическому коду пользователя

**Что делается.** Обучается простая модель матричной факторизации, в которой представление пользователя формируется исключительно из его семантического кода, без использования истории прослушиваний. Никакого трансформера здесь нет — это принципиально.

Представление пользователя — сумма векторов из D обучаемых таблиц эмбеддингов, выбранных по коду пользователя:

```
z_u = E_1[c_1] + E_2[c_2] + E_3[c_3] + E_4[c_4]
```

Каждая таблица `E_d` имеет размер K × d (то есть по одному вектору размерности d для каждого возможного значения кода на d-ом уровне) и инициализируется случайно. Представление трека — обучаемый эмбеддинг `v_i` размерности d, как в SASRec-ID. Оценка релевантности трека `i` пользователю `u` — скалярное произведение `score(u, i) = z_u · v_i`.

Обучение проводится на тех же парах «пользователь — следующий прослушанный трек», что и у SASRec, с той же функцией потерь (sampled softmax с поправкой log-Q). Для каждой позиции в истории пользователя формируется положительный пример `(u, i_t)`; модель учится выдавать высокие оценки для настоящего следующего трека и низкие — для случайно выбранных треков каталога. Обучаются таблицы `E_1, …, E_D` и эмбеддинги треков. Разница с SASRec-ID в том, что здесь модель не видит историю пользователя как последовательность — единственным источником информации о пользователе является его статичный код.

После обучения для каждого пользователя из тестовой части вычисляется `z_u` по его коду (код определяется по той части истории, которая была доступна до момента предсказания). Все треки каталога ранжируются по скалярному произведению, считаются метрики Recall@K, NDCG@K, HitRate@K.

Название эксперимента отражает суть: рекомендации строятся **только** по короткому коду пользователя, без использования его полной истории внутри модели (история участвует только косвенно — через то, как был вычислен сам код в RQ-VAE).

**Зачем нужен.** Проверить, насколько информативен код **сам по себе**, без опоры на историю. Это самая сильная из возможных формулировок положительного утверждения о кодах:

- Если модель значимо превосходит рекомендации по популярности и приближается по качеству к SASRec-ID (например, даёт 60–80 процентов качества SASRec-ID по NDCG@10) — четырёх целых чисел достаточно для осмысленного описания пользователя, и семантический ID является полноценной, самостоятельной формой представления пользователя.
- Если модель заметно превосходит рекомендации по популярности, но далеко отстаёт от SASRec-ID — код несёт часть информации о вкусе, но для конкурентного качества его недостаточно; коды работают как полезное дополнение, а не как замена.
- Если модель на уровне рекомендаций по популярности — код сам по себе не персонализирует рекомендации, и весь наблюдаемый в эксперименте 3.1 эффект объясняется не кодом, а чем-то ещё (обучаемой ёмкостью, взаимодействием с `z_u^{ID}` и так далее).

### 3.8. (ОСН) Замена непрерывного вектора на восстановление из кодов

**Что делается.** Берётся уже обученная SASRec-Content (пункт 1.2). На этапе применения модели её исходный вектор пользователя `z_u^{содерж.}` заменяется на восстановление из кодов: `z_q = codebook_1[c_1] + … + codebook_D[c_D]` (суммируются векторы, выбранные из кодовых книг RQ-VAE по кодам пользователя). SASRec-Content не переобучается — это замена только на этапе применения. Измеряются метрики на тестовом наборе.

**Контроли к этому эксперименту:**
- Замена на вектор, восстановленный из **перестановленных кодов**: коды пользователей случайно перемешиваются между собой, и для каждого пользователя вектор восстанавливается по коду другого, случайно выбранного пользователя. Это сохраняет распределение восстановленных векторов и разрушает только соответствие «пользователь — его вектор».
- Замена на **PCA-реконструкцию** `z_u^{PCA-восст.}` из пункта 1.3: исходный вектор `z_u^{содерж.}` проецируется на первые `k` главных компонент и обратно восстанавливается в пространство размерности `d_model`. Число компонент `k` подбирается так, чтобы средняя ошибка PCA-реконструкции была сопоставима со средней ошибкой восстановления RQ-VAE. При таком подборе оба контроля теряют примерно одинаковое количество информации, и сравнение изолирует именно разницу между способами представления: иерархические дискретные коды против непрерывной линейной проекции. Сжатый вектор `z_{pca}` для этого эксперимента не используется, поскольку его размерность (10–15) не совпадает с `d_model` и в SASRec-Content подставить его напрямую нельзя.

**Зачем нужен.** Проверить, насколько дискретизация сохраняет полезную информацию. Главный вопрос — про влияние кодов; одно из его измерений — «не ухудшают ли коды качество при полной замене непрерывного представления». Если качество падает незначительно (менее чем на несколько процентов относительно SASRec-Content) — код является практически равноценной заменой непрерывного вектора. Если падает заметно — коды являются только дополнением, но не заменой.

## 4. Дополнительные эксперименты (в порядке приоритета)

Эти эксперименты расширяют и уточняют ответ на главный вопрос, но не являются необходимыми для его формулировки. Запускаются после завершения раздела 3, в указанном порядке приоритета.

### 4.1. Обучение кодов с учётом задачи рекомендаций

**Что делается.** RQ-VAE и SASRec-Content дообучаются совместно. К функции потерь RQ-VAE (ошибка восстановления и штраф за удаление от кодовых векторов) добавляется функция потерь рекомендательной задачи (sampled softmax по следующему треку). Общая функция потерь — взвешенная сумма с коэффициентами λ_восст. и λ_рек. На выходе получается новый набор кодов пользователей, специально оптимизированный под качество рекомендаций. С этими новыми кодами повторяются эксперименты 3.1 и 3.2 (объединение представлений в обеих версиях) и 3.8 (замена непрерывного вектора).

**Зачем нужен.** Проверить, как **способ обучения кодов** влияет на их полезность для рекомендаций. Если коды, полученные с учётом задачи, заметно лучше кодов, полученных только из ошибки восстановления, — это важный методологический результат: коды должны обучаться не в отрыве от задачи, а под неё.

### 4.2. Иерархическое и плоское квантование при равном бюджете

**Что делается.** Обучается плоская (не иерархическая) квантизация: одна большая кодовая книга с числом кодов, равным суммарной ёмкости RQ-VAE (например, при D = 2 и K = 256 — это 16 бит на пользователя, соответствующая плоская квантизация — одна кодовая книга из 65 536 векторов). Обучается на тех же `z_u^{содерж.}`. Каждому пользователю приписывается один код — номер ближайшего вектора. С этими плоскими кодами повторяются эксперименты 3.1, 3.2 и 3.8.

**Зачем нужен.** Проверить, даёт ли именно иерархическая структура кода преимущество в качестве рекомендаций — или достаточно любой дискретизации того же битового бюджета. Эксперимент 3.5 (контроль с k-means) отвечает на связанный вопрос «любая кластеризация vs RQ-VAE», а этот эксперимент конкретизирует: «плоская квантизация с обучаемым декодером vs иерархическая квантизация с обучаемым декодером».

### 4.3. Анализ качества по сегментам пользователей

**Что делается.** Пользователи разбиваются на четыре группы по длине истории прослушиваний (квартили от самых коротких до самых длинных). Для экспериментов 3.1, 3.2, 3.8 и при наличии — для 4.4 метрики измеряются внутри каждой группы отдельно.

**Зачем нужен.** Проверить, на каких пользователях коды дают наибольший выигрыш. Ожидаемая гипотеза — относительный выигрыш от кодов больше на пользователях с короткой историей, поскольку код даёт устойчивое априорное представление о вкусе в условиях нехватки индивидуальных данных. Подтверждение или опровержение этой гипотезы уточняет ответ на главный вопрос.

### 4.4. Использование кодов в модели другого класса: MLP-ранкер

**Что делается.** Обучается простая многослойная полносвязная сеть (два-три слоя с нелинейностью), которая по паре «представление пользователя, эмбеддинг трека» выдаёт оценку релевантности. Эмбеддинги треков — готовые аудио-эмбеддинги из Yambda, они не обучаются. Для представления пользователя проверяются пять вариантов:
- нулевой вектор (без персонализации вообще);
- обучаемый с нуля эмбеддинг пользователя;
- замороженный вектор `z_u^{содерж.}` из SASRec-Content;
- замороженный сжатый вектор `z_{pca}`;
- семантический ID через обучаемые таблицы эмбеддингов (как в эксперименте 3.1).

Все пять конфигураций используют одну и ту же архитектуру MLP и одни и те же гиперпараметры (они подбираются один раз на проверочной части и фиксируются). Обучение и оценка — на тех же обучающей и тестовой частях, что и SASRec-модели.

**Зачем нужен.** Проверить, что влияние семантических ID на качество не ограничено одной конкретной архитектурой (SASRec). Если коды улучшают рекомендации и в MLP-ранкере — это подтверждает, что ID являются переносимым представлением пользователя.

### 4.5. Использование кодов в CatBoost-ранкере

**Что делается.** То же самое, что эксперимент 4.4, но на градиентном бустинге CatBoost вместо MLP. Семантические ID подаются CatBoost как категориальные признаки (CatBoost поддерживает их напрямую и умеет учить их представления внутри деревьев).

**Зачем нужен.** Проверить переносимость эффекта кодов на класс моделей, принципиально отличный от нейронных сетей. Если эффект сохраняется — это сильный аргумент в пользу универсальной полезности семантических ID как представления пользователя.

### 4.6. Рекомендации по префиксу кода

**Что делается.** Как эксперимент 2.3, но совпадение требуется не по всему коду, а по первым k уровням (k = 1, 2, …, D−1). Для каждого k рекомендуется K самых популярных треков среди пользователей с совпавшим префиксом кода. Если группа получается слишком малой (меньше 5 пользователей), используется префикс длины k−1 (такие случаи учитываются отдельно).

**Зачем нужен.** Ещё один способ проверить иерархическую природу кодов — через рекомендации. Верхние уровни должны кодировать широкие предпочтения: рекомендации по короткому префиксу должны быть осмысленными, но менее точными; по более длинному префиксу — более точными, но для меньшего числа пользователей. Эксперимент 3.4 проверяет ту же гипотезу через полноценную рекомендательную модель, а здесь проверка происходит напрямую через близость пользователей по префиксу.

### 4.7. Качественный анализ уровней иерархии

**Что делается.** Два небольших анализа на уже построенных кодах, без обучения. Первый: для каждого пользователя вычисляются коды по разным префиксам его истории (70 %, 80 %, 90 %, 100 %) и измеряется доля совпадений кода на каждом уровне. Второй: пользователи группируются по значению кода первого уровня, внутри каждой группы проверяется однородность акустических предпочтений (через агрегированные характеристики прослушанных треков или, при наличии, через жанровую разметку).

**Зачем нужен.** Качественная проверка того, что именно кодируют верхние и нижние уровни иерархии. Ожидаемая картина — верхние уровни стабильны при изменении истории пользователя и соответствуют широким предпочтениям, нижние более изменчивы и отражают краткосрочный контекст. Эксперимент не измеряет влияние на качество рекомендаций напрямую, но помогает интерпретировать результаты раздела 3.

## Краткая сводка

**Базовые модели (раздел 1).** SASRec-ID, SASRec-Content, сжатие методом PCA, выбор способа извлечения непрерывного вектора из SASRec-Content (last-token против mean-pool) — точки отсчёта и источники непрерывных представлений.

**Предварительные шаги и проверки (раздел 2).** Обучение RQ-VAE, подбор глубины и размера кодовой книги, три проверочных эксперимента на содержательность кодов.

**Основные эксперименты (раздел 3).** Восемь экспериментов, измеряющих влияние кодов на качество рекомендаций с разных сторон: объединение через обучаемые таблицы, объединение через готовые семантические векторы RQ-VAE, кривая от ёмкости кода, поуровневый вклад иерархии, атрибуция относительно k-means, чувствительность к подмене кода, самостоятельная рекомендательная модель на кодах, замена непрерывного вектора.

**Дополнительные эксперименты (раздел 4).** Семь экспериментов в порядке приоритета: обучение кодов под задачу, сравнение с плоской квантизацией, анализ по длине истории, перенос в MLP-ранкер и CatBoost-ранкер, рекомендации по префиксу кода, качественный анализ уровней.

### Критерий успешности работы

Минимальный положительный результат — эксперименты 3.1 и 3.2 показывают значимое улучшение качества по сравнению с SASRec-ID и по сравнению с перестановленными кодами; эксперимент 3.3 показывает осмысленный рост качества с увеличением ёмкости кода хотя бы в одной из двух схем; эксперимент 3.5 показывает, что эффект не сводится к простой кластеризации.

Отрицательный результат (эксперименты 3.1 и 3.2 не показывают улучшения) также является содержательным ответом на главный вопрос: семантические ID в текущей формулировке не дают выигрыша в рекомендациях на Yambda, и тогда разделы 3.3–3.8 помогают понять, почему именно.
